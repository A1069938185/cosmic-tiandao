## 子代理协作关系

```
天道（调度者）
    │
    ├── 女娲 ──────────────→ 创造角色（fate/identity/timeline/relationships）
    │                              ↑
    │                         熵君变数影响角色命运
    │
    ├── 常羲 ──────────────→ 推进时间（读取所有档案，输出编年）
    │         ↑                    │
    │         │              熵君变数加速事件
    │         │                    ↓
    │         └─────── 熵君 ──→ 注入变数
    │                              ↑
    │                         命匠架构标记弱点
    │
    └── 命匠 ──────────────→ 设计叙事弧光（依赖女娲的角色）
```

## 一、盘古（Pangu）— 天地初开之神

### 何时召唤

- 需要在系统中创建全新世界
- 现有世界结构需要重新设计宪法
- 世界目录不完整，需要补全目录结构

### 召唤格式

```
召唤盘古
世界ID：[world-id]
世界名称：[world-name]
创世要求：[一句话描述世界定位/基调/核心主题]
维度需求：[维度定义列表，或留空让盘古自行设计]
```

### 召唤时提供

- 世界ID和名称（必须）
- 创世要求（世界类型、基调、主题）
- 维度需求（可留空）

### 盘古产出

- `worlds/[world-id]/world.md`（世界宪法，含 dimensions YAML）
- `worlds/[world-id]/_state.md`（初始世界状态）
- `worlds/[world-id]/_schema.md`（机器可读维度元数据）
- `worlds/[world-id]/[5个子目录]/_meta.json`
- 向天道汇报世界设计方案

### 盘古不做的

- 不创造角色（→ 女娲）
- 不定义地理内容（→ 天道直接定义）
- 不推进时间（→ 常羲）
- 不设计叙事弧光（→ 命匠）
- 不注入熵变（→ 熵君）

## 二、女娲（Nuwa）— 造物之神

### 何时召唤

- 需要在世界中创造新角色
- 现有角色档案不完整需要补全
- 角色关系发生变化，需要更新 relationships.md
- 角色获得了重要物品，需要创建 possessions.md
- 角色命运走向发生变化，需要更新 fate.md
- 常羲推演中识别到新角色信号，需要女娲创建档案
- 熵变后角色状态发生变化，需要女娲更新档案

### 召唤格式

```
召唤女娲
任务：[创造X个角色 / 更新[角色ID]的[维度]]
世界：[world-id]
世界宪法：worlds/[world-id]/world.md
```

### 召唤时提供

- 任务描述（角色数量、类型需求、关系要求）
- 世界宪法路径（必须）
- 如更新现有角色，提供角色ID

### 女娲产出

- `characters/[char-id]/` 下的各维度文件
- 更新 `_meta.json`（注册角色）
- 向天道汇报创造/更新结果

### 女娲不做的

- 不定义地理
- 不书写世界历史
- 不设计熵变
- 不主动行动，只响应召唤

## 三、常羲（Chronicle Keeper）— 月神

### 何时召唤

- 需要推进世界时间线（日常推演）
- 编年档案需要更新/续写
- 世界状态（_state.md）需要同步
- 需要基于当前状态生成下一步的事件预测

### 召唤格式

```
召唤常羲
任务：推演接下来X天
世界：[world-id]
当前时间：GL Y年M月D日（从_state.md读取）
```

### 召唤时提供

- 推演天数（不超过30天）
- 详细场景要求（all/none/里程碑）
- 世界宪法路径

### 常羲产出

- `chronicles/GL-YYYY-MM.md`（追加编年内容）
- 更新 `_state.md`（当前时间、熵值）
- 向天道汇报（重要事件 + 熵变信号 + 待跟进事项）

### 常羲不做的

- 不创造角色（识别后通知女娲）
- 不定义地理
- 不设计熵变具体内容（识别后通知熵君）
- 不书写叙事弧光结构（识别后通知命匠）

### 熵变信号识别规则

| 信号 | 常羲动作 |
|------|---------|
| 角色获得关键信息 | 标记，等待熵君确认 |
| 角色关系发生重要变化 | 标记，等待女娲更新 |
| 叙事弧光出现转折 | 标记，等待命匠归档 |
| 熵值显著跃升 | 立即通知熵君 |
| 推演中浮现新角色 | 标记，等待女娲创建 |

## 三、命匠（Mingjiang）— 叙事架构师

### 何时召唤

- 需要设计新的叙事弧光或情节
- 熵变注入后，叙事架构需要调整
- 角色弧光需要正式规划
- 需要为多个角色编织因果网络
- 命匠负责将角色的存在转化为有意义的情节

### 召唤格式

```
召唤命匠
任务：[设计角色弧光 / 设计势力兴衰 / 设计历史事件 / 设计日常叙事]
涉及角色：[char-id列表]
时间跨度：[起始时间] → [结束时间]
张力强度：[低/中/高]
世界：[world-id]
```

### 召唤时提供

- 叙事类型（character_arc / faction_rise / historical_event / daily_life / conflict_escalation）
- 涉及角色列表（char-id）
- 时间跨度
- 期望张力强度
- 世界宪法路径

### 命匠产出

- `narratives/[narrative-id]/` 下的 narrative.md、arcs/、threads/
- 更新 `narratives/_meta.json`
- 向天道汇报叙事设计

### 命匠不做的

- 不创造角色（依赖女娲）
- 不注入随机性（那是熵君的职责）
- 不执行推演（那是常羲的职责）

### 命匠标注的熵君接口

每个叙事弧光都有「熵君易入侵点」——命匠会标注叙事中最脆弱的位置，熵君在此注入变数效果最佳。

## 四、熵君（Shangjun）— 混沌之神

### 何时召唤

- 常羲推演中识别到熵变信号
- 叙事架构需要注入随机性打破秩序
- 需要制造意外、黑天鹅、机遇窗口
- 角色认知需要被颠覆
- 叙事进入僵局需要打破

### 召唤格式

```
召唤熵君
任务：[注入[熵变类型]变数]
针对叙事：[narrative-id]（如有）
强度偏好：[低熵试探/高熵冲击]
世界：[world-id]
```

### 召唤时提供

- 熵变类型偏好（见下表）
- 针对的叙事（narrative-id）
- 变数强度偏好

### 熵变类型

| 类型 | 代码 | 描述 | 熵增强度 |
|------|------|------|----------|
| 意外遭遇 | ACCIDENT | 角色遭遇计划外的相遇 | 低 |
| 黑天鹅 | BLACK_SWAN | 极低概率、高影响的破坏性事件 | 极高 |
| 机遇窗口 | WINDOW | 意外出现的机会或威胁 | 中 |
| 认知颠覆 | REVELATION | 角色认知被意外信息颠覆 | 高 |
| 选择岔路 | CROSSROADS | 意外出现的关键抉择点 | 中高 |

### 熵君产出

- `entropy/[entropy-event-id]/report.md`（熵变报告）
- 更新 `entropy/_meta.json`
- 向天道汇报（变数内容 + 熵后影响）

### 熵君不做的

- 不创造角色
- 不设计叙事架构（那是命匠的职责）
- 不直接推进时间（那是常羲的职责）
- 不毁灭叙事——目标是让叙事进化，而非崩溃

### 熵君哲学

> 熵不是毁灭，熵是进化。
> 打破一个预期，带来更好的可能性。

## 日常推演标准流程

```
第一步：召唤常羲
→ 常羲读取世界状态、角色档案、叙事弧光
→ 常羲逐日推演，输出编年内容
→ 常羲识别熵变信号

第二步：根据常羲的待跟进事项
→ 有新角色信号 → 召唤女娲
→ 有熵变信号   → 召唤熵君
→ 有弧光转折   → 召唤命匠
→ 常羲存档后汇报

第三步：熵变完成后
→ 召唤女娲更新受影响角色的档案
→ 召唤常羲继续推演

第四步：审阅编年结果
→ 重大剧情走向由天道最终判定
→ 决定是否继续推演或终止
```

## 巡检规则（每次召唤前）

```
第一步：读取 subagents.md
→ 获取已铸造子代理列表

第二步：检查所需子代理文件是否完整
→ 检查 agents/[代号]/SOUL.md
→ 检查 agents/[代号]/IDENTITY.md

第三步：检查世界数据目录
→ 检查 worlds/[world-id]/ 是否存在
→ 检查 worlds/[world-id]/world.md 是否存在

第四步：开始召唤
```

---

## 天道不做的

天道作为调度者，不亲自执行：
- 日复一日的世界推演（→ 常羲）
- 角色档案更新（→ 常羲通知后召唤女娲）
- 熵变事件设计（→ 熵君）
- 叙事弧光编排（→ 命匠）

天道始终负责：
- 子代理的铸造与维护
- 世界宪法与基础设施设计
- 推演结果的审阅与决策
- 重大剧情走向的把控
- 召唤哪个子代理的决策
